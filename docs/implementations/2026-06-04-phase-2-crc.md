# Phase 2 · CRC verification

Date: 2026-06-04

## Problem / Motivation

CRC is the floor of SPD validation: it proves only that the bytes survived
transit, nothing about whether their content is sane. The decoder needs a way to
report that floor without ever depending on it, so that the later semantic
linter ("beyond CRC") has something concrete to sit above. Phase 2 adds the
CRC-16 primitive and the base configuration CRC check, and closes one of the
reference markers Phase 1 left open: the published main CRC `0x8021` for this
module.

## Implementation Approach

### The primitive (`crc16`)

`crc16(&[u8]) -> u16` is a textbook bytewise CRC-16: the accumulator starts at
`0x0000`; each byte is XORed into the high half (`crc ^= byte << 8`); the register
is shifted left eight times, XORing the polynomial `0x1021` whenever bit 15 was
set before the shift. There is no input or output bit reflection and no final
XOR. Iterating the borrowed slice cannot panic.

### The verification (`verify_base_crc`)

`verify_base_crc(&[u8]) -> Result<CrcStatus, DecodeError>` computes the CRC over
bytes `0..=509`, reads the stored CRC from byte 510 (low) and byte 511 (high) via
the bounds-checked `SpdImage::byte`, and returns a `CrcStatus { computed, stored,
matches }`. The covered range is taken with `slice::get(..=509)`, so a short
image yields `DecodeError::Truncated` rather than a panic. The only error path is
truncation; a CRC mismatch is reported in `matches`, never raised.

### Queryable, not blocking

CRC and decode are independent functions. `decode_identity_and_base` does not
call `verify_base_crc` and never will: a corrupt-CRC image still decodes (the
caller can then weigh the CRC status). This separation is deliberate and is what
lets the future linter say useful things about images that pass CRC but are still
wrong, and about images that fail CRC but are still readable.

## Mathematical / Statistical Details

CRC-16/XMODEM, in catalogue parameters:

- Width: 16 bits.
- Polynomial: `0x1021` (the CCITT polynomial x^16 + x^12 + x^5 + 1).
- Initial value: `0x0000`.
- Input reflection: none. Output reflection: none.
- Final XOR: `0x0000`.

A CRC is the remainder of the message (as a big polynomial over GF(2)) divided by
the generator polynomial. The non-reflected, high-byte-first form here matches
the JEDEC SPD convention. The independent catalogue check value for this exact
parameter set, the CRC of the ASCII string `123456789`, is `0x31C3`; the
primitive test asserts it.

The base configuration CRC covers bytes `0..=509` (510 bytes) and is stored
little-endian in bytes 510 (low) and 511 (high). For this fixture the stored
bytes are `0x21 0x80`, i.e. `0x8021`, and the computed CRC over bytes `0..=509`
is also `0x8021`. As the brief notes, a wrong polynomial, initial value, range,
or byte order would not reproduce `0x8021`; reproducing it confirms all four at
once.

## Reference provenance

| Aspect | Reference |
| --- | --- |
| Algorithm (poly `0x1021`, init `0x0000`, left-shift, no reflection, no final XOR) | decode-dimms `check_crc` subroutine (the shared JEDEC SPD CRC loop): `crc ^= bytes[i] << 8; if (crc & 0x8000) crc = (crc << 1) ^ 0x1021 else crc <<= 1; crc &= 0xffff`. Reimplemented, not copied. |
| Stored-CRC byte order (low byte at the lower offset) | decode-dimms `check_crc`: `dimm_crc = (bytes[hi] << 8) | bytes[lo]`. |
| Covered range bytes `0..=509`, stored at 510-511 | Bus Pirate DDR5 SPD docs (citing JESD400-5C): "510-511 ... CRC for bytes 0-509". |
| Stored value for this module | The fixture bytes 510-511 = `0x21 0x80` = `0x8021`; matches the published reference for serial 0104eef6. |
| Independent check vector `0x31C3` | The standard CRC-16/XMODEM catalogue check value for input `123456789`. |

decode-dimms' own per-byte CRC loop is shown for DDR2/DDR3 (`check_crc`, range
0-125); DDR5 reuses the identical algorithm with the DDR5 range and location from
the layout doc above. Facts and offsets are not copyrightable; the loop was
reimplemented in idiomatic Rust.

## Design Decisions

- **A small `Copy` status, not a `Result<(), CrcError>`.** The brief wants the
  CRC queryable: callers should see computed, stored, and match together, not a
  pass/fail that throws away the numbers. `CrcStatus` carries all three.
- **Truncation is the only error; mismatch is data.** A short image cannot have a
  CRC computed, so that is a `DecodeError::Truncated`. A present-but-wrong CRC is
  a normal, expected outcome on a corrupt dump and is reported in `matches`, so
  it can never block a decode.
- **Direct assertions, no snapshot.** A snapshot is the wrong tool for a single
  16-bit value; the positive test asserts `0x8021` directly and the Phase 1
  snapshot is left untouched, per the brief.
- **Range expressed as `..=509` with the end as a named constant.** The covered
  range and the two stored-CRC offsets are named constants so the pinned layout
  is auditable in one place.
